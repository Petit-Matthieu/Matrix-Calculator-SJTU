02 · METHOD 01 · 原始三重循环

METHOD 01 · NAIVE I-J-K

原始三重循环

循环顺序 i → j → k，直接翻译矩阵乘法定义：

C[i][j] = Σ A[i][k] × B[k][j]

内存访问流程
A[i][k]
→ 沿行连续访问 · cache hit
B[k][j]
→ 沿列跳跃 6144B · cache miss 每次
C[i][j]
→ 读-改-写 · 内存往返
耗时
564.9ms
加速比
1×
// MatrixMultiplyOriginal — i-j-k 三重循环
// i 遍历 A 的每一行
for (int i = 0; i < A->row; ++i) {
  // j 遍历 B 的每一列
  for (int j = 0; j < B->column; ++j) {
    REAL sum = 0.0;
    // k 累加求和 — B 按列跳跃！
    for (int k = 0; k < A->column; ++k) {
      sum += A->data[MatrixIndex(A, i, k)]
          * B->data[MatrixIndex(B, k, j)];
      //                    ↑ 跳整行
    }
    C->data[MatrixIndex(C, i, j)] = sum;
  }
}
// 问题：内层 k 循环中
// B[k][j] → B[k+1][j] 跳跃 768×8 = 6144 字节
// 远超 64 字节缓存行 → 每次 cache miss
// 延迟从 ~4 周期暴涨到 ~200 周期

METHOD 02 · I-K-J REORDER

循环重排

将循环顺序从 i-j-k 改为 i → k → j，内层循环沿 j 方向（行方向）遍历 B 矩阵。

访问模式对比
BEFORE · i-j-k
B[0][j], B[1][j]...
列跳跃 · 每次 miss
AFTER · i-k-j
B[k][0], B[k][1]...
行连续 · 全部 hit
两个关键优化

① B 按行访问：相邻元素仅差 8 字节，落在同一条 64B 缓存行内，一次加载服务 8 次访问。

② aik 标量复用：提取为局部变量后保留在寄存器中，内层循环省去 A 的索引开销。

耗时
~215ms
加速比
2.6×
// MatrixMultiplyIKJ — i-k-j 循环重排
for (int i = 0; i < A->row; ++i) {
  // k 提到 j 前面！
  for (int k = 0; k < A->column; ++k) {
    // aik 提取为标量，保留在寄存器
    REAL aik = A->data[MatrixIndex(A, i, k)];
    // j 内层循环 — B 按行连续访问
    for (int j = 0; j < B->column; ++j) {
      C->data[MatrixIndex(C, i, j)]
        += aik * B->data[MatrixIndex(B, k, j)];
      //                     ↑ 行连续
    }
  }
}
// 效果：B[k][j] → B[k][j+1] 仅差 8 字节
// 落在同一条 64B 缓存行内
// 一个缓存行服务 8 次连续访问
// cache hit 率接近 100%

// 局限：C 矩阵大时仍会 cache thrashing
02 · METHOD 03 · 缓存分块
08 / 14
METHOD 03 · BLOCKED + UNROLL

64×64 分块 + 4 路展开

768×768 矩阵有 4.7MB，但 CPU 的高速缓存装不下。切成 64×64 小块，每次只算一小块，数据一直在高速缓存里不搬走。

L1 缓存是什么？为什么这么重要？

CPU 内部有三级缓存，越靠近 CPU 越快但越小：

• L1 缓存：32-48KB，访问约 4 个时钟周期，最快

• L2 缓存：256KB-1MB，访问约 12 个周期

• 主内存（RAM）：几 GB，访问约 200 个周期，比 L1 慢 50 倍

64×64 个 double = 32KB，刚好塞进 L1。算这一小块时所有数据都在"快车道"上，不用去"慢车道"取。这就是为什么分块后速度能快 10 倍。

为什么偏偏是 4 路展开，不是 2 或 8？

正常循环：算 1 个 → 判断"还要继续吗" → 算 1 个 → 判断 → ...

4 路展开：一口气算 4 个 → 判断 → 算 4 个 → ...

选 4 的三个原因：

① CPU 的浮点乘法器通常有 2-4 个，4 路刚好打满，再多了也不会更快

② 4 个 double = 32 字节，对齐友好，CPU 读取效率最高

③ 展开太多（比如 16 路）代码体积膨胀，反而挤占指令缓存，得不偿失

实测 2 路、4 路、8 路，4 路是性价比最高的甜点。

耗时
~53.5ms
加速比
10.5×
// MatrixMultiplyBlocked — 分块 + 4 路展开
#define BLOCK_SIZE 64
// 外层：按 64×64 分块
for (ii = 0; ii < N; ii += BLOCK_SIZE)
 for (kk = 0; kk < N; kk += BLOCK_SIZE)
  for (jj = 0; jj < N; jj += BLOCK_SIZE)
   // 块内：i-k-j 循环
   for (i = ii; i < i_end; ++i)
    for (k = kk; k < k_end; ++k) {
     REAL aik = a_row[k];
     // j 方向 4 路展开
     for (j = jj; j+3 < j_end; j += 4) {
      c_row[j]   += aik * b_row[j];
      c_row[j+1] += aik * b_row[j+1];
      c_row[j+2] += aik * b_row[j+2];
      c_row[j+3] += aik * b_row[j+3];
     }
     for (; j < j_end; ++j) // 尾部
      c_row[j] += aik * b_row[j];
    }
// 64×64 块 = 32KB → 适配 L1
// 4 路展开 → 减少 75% 循环开销

// 局限：内层仍对 C 做读-改-写
// 每次迭代 2 次内存往返

METHOD 04 · 4×4 MICROKERNEL

微内核 + 分块

上一步虽然分块了，但每次算 C 的一个元素还是要"从内存读出来→加上去→写回内存"，反复折腾。

这一步：用 16 个局部变量代替 C 矩阵，全程在 CPU 寄存器里算，768 轮循环结束后才写回一次。

两个核心概念
寄存器是什么？
CPU 内部最快的存储，比 L1 缓存还快 10 倍。容量极小（只有几十个），但访问零延迟。编译器会自动把局部变量放进去。
restrict 是什么？
C 语言的一个关键字，放在指针前面。意思是"我保证这块内存跟别人不重叠"。没有它，编译器怕你改 A 会影响到 B，不敢把变量留在寄存器里。加了 restrict，编译器就放心大胆优化。
实际效果

768 次循环 × 16 个元素 × 读+写 = 24576 次内存操作 → 16 次

B 的一行 4 个数被 A 的 4 行共享，读一次用 4 遍

耗时
30.3ms
加速比
18.66×

// Multiply4x4 — 每次算 C 的一个 4×4 小块
static void Multiply4x4(
  const REAL * restrict a, int a_stride,
  const REAL * restrict b, int b_stride,
  REAL * restrict c, int c_stride,
  int inner) {
  // restrict = "这块内存跟别人不重叠，放心优化"
  // 先搞 16 个局部变量，代替 C 矩阵
  // 这些变量会被 CPU 放进寄存器（最快的存储）
  REAL c00=0, c01=0, c02=0, c03=0;
  REAL c10=0, c11=0, c12=0, c13=0;
  REAL c20=0, c21=0, c22=0, c23=0;
  REAL c30=0, c31=0, c32=0, c33=0;
  // 沿内维 k 逐层累加
  for(k=0; k<inner; ++k) {
   // 取 B 的一行 4 个数
   b0=b[k][0]; b1=b[k][1]; b2=b[k][2]; b3=b[k][3];
   // 取 A 的 4 行各 1 个数
   a0k=a0[k]; a1k=a1[k]; a2k=a2[k]; a3k=a3[k];
   // 4×4=16 次乘加，全在寄存器里完成
   // B 的 4 个数被 A 的 4 行共享（读一次用 4 遍）
   c00+=a0k*b0; c01+=a0k*b1; c02+=a0k*b2; c03+=a0k*b3;
   c10+=a1k*b0; c11+=a1k*b1; c12+=a1k*b2; c13+=a1k*b3;
   c20+=a2k*b0; c21+=a2k*b1; c22+=a2k*b2; c23+=a2k*b3;
   c30+=a3k*b0; c31+=a3k*b1; c32+=a3k*b2; c33+=a3k*b3;
  }
  // 循环跑了 768 轮之后，才把 16 个结果写回 C
  // 之前全程没碰过 C 矩阵的内存！
  c[0]=c00; c[1]=c01; c[2]=c02; c[3]=c03; c+=stride;
  c[0]=c10; c[1]=c11; c[2]=c12; c[3]=c13; c+=stride;
  c[0]=c20; c[1]=c21; c[2]=c22; c[3]=c23; c+=stride;
  c[0]=c30; c[1]=c31; c[2]=c32; c[3]=c33;
}

WHAT IS APPLE BLAS?

BLAS（Basic Linear Algebra Subprograms）是线性代数运算的工业标准接口。Apple Accelerate 是苹果官方的高性能数学库，内部使用 AMX 矩阵协处理器 + 多线程 + SIMD 向量化。

它快是因为用了硬件专用指令，而我们的代码是纯 C 软件实现。差距 18× 说明：硬件加速 vs 通用 CPU 的天花板差距就是这么大。

SOLVING Ax = b

LU 分解四步法

STEP 01 LU 分解 A = L·U
无主元选取 O(n³)
STEP 02 前向代入 解 Ly = b
逐行前推 O(n²)
STEP 03 回代 解 Ux = y
逐行回推 O(n²)
STEP 04 行列式 det(A) = ∏Uᵢᵢ
对角线乘积 O(n)
数值说明：LUDecomposeNoPivot 有意实现为不选主元的 LU 分解，适用于教学和输入受控的场景。若需数值稳定性，应使用带主元选取的 LAPACK 分解。
